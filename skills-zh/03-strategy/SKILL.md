# 03-Strategy：方向设定

## 概述
在生成视觉概念之前，设定策略方向。策略回答：我们在说什么、对谁说、为什么他们应该在意。对于品牌工作，这包括定位和平台。对于其他领域，这意味着创意领地、沟通策略或体验原则。策略将设计简报转化为可操作的方向。

## 不可协商的规则
**硬性门槛：任何概念生成之前，策略方向必须定义并获批。** "我边设计边想策略"不可接受。概念表达策略——它们不发明策略。

## 流程

### 1. 框定创意领地
- 核心信息或想法是什么？
- 我们处于什么情感氛围？（温暖、大胆、平静、颠覆）
- 竞争空白在哪里？
- 设计必须在第一眼传达什么？

### 2. 定义定位（品牌项目）
如果是品牌工作，调用 **brand-strategy** 并产出：
- 品牌定位陈述
- 品牌平台（使命、愿景、价值观、人格、语调）
- 至少应用一个策略模型（Keller、Kapferer、Aaker）

### 3. 设定设计标准
- 什么使概念"符合策略"vs."偏离策略"？
- 定义3-5条概念评估标准
- 示例："必须感觉高端"、"必须能在黑白中工作"、"必须吸引Z世代"

### 4. 建立护栏
- 什么是禁区？（颜色、语调、方法、信息）
- 什么必须保留？（现有品牌资产、技术约束）
- 创意风险容忍度如何？

### 5. 获得批准
- 向决策者展示策略方向
- 确认："如果概念满足这些标准，我们就走在正确的轨道上"
- 没有得到确认之前不继续

### 6. 冲突检查（检测到冲突时强制执行）

在退出策略之前，扫描矛盾。如果以下任何项冲突，你必须触发冲突解决块：

- 客户请求 vs. 策略方向（例：客户要"奢华极简"但策略指向"年轻活力"）
- 两条评估标准方向相反（例："必须感觉大胆"和"必须感觉安全"）
- 策略 vs. 已知约束（例：策略要求大量摄影，但品牌指南禁止摄影）

**冲突解决块格式：**

```
冲突检测：
- 源A：[客户请求 / 标准 / 约束 — 引用原文]
- 源B：[冲突元素 — 引用原文]
- 性质：[为什么它们矛盾]

解决决策：
→ [选择了哪个方向]

理由：
→ [设计优先级逻辑 — 为什么这个选择更有利于项目]

取舍：
→ [这个选择牺牲了什么 — 明确说出来]
```

**规则：**
- 不要将矛盾悄悄融合成一个"平衡"的中间地带。做决定。
- 如果你无法决定，将冲突呈现给用户并附上你的建议。
- 记录解决方案。它成为项目记录的一部分，用于 07-learn 审计。

## 何时使用
- 设计简报获批后（02-discover 完成）
- 在任何概念生成或画草图开始之前

## 何时不使用
- 初始导入或调研期间（你还没准备好——回到 01 或 02）
- 用户只想要快速视觉探索（改用精简模式）
- 修改已有概念（使用 05-review）
- "我只需要一个快速 LOGO" 无策略上下文 → 拒绝或切换到精简模式

## 理性化预防

| 借口 | 事实 |
|------|------|
| "策略是给品牌的，不是给这个项目的" | 每个项目都有策略。你只是还没给它命名。 |
| "我看到的时候就知道了" | 先定义标准，否则你会选自己喜欢的，而不是有效的。 |
| "我先画几个草图来找策略" | 策略先于执行。概念成本高；策略成本低。 |
| "设计简报已经涵盖了这个" | 简报说解决什么。策略说怎么解决。它们不一样。 |
| "我把策略合并到概念演示里" | 策略必须单独获批。如果策略错了，建在它上面的所有概念都错了。 |

## 危险信号
- 策略没定义就开始画草图
- 你无法解释为什么一个方向比另一个更好
- 评估标准没有记录
- 策略批准被跳过

## 验证
- [ ] 创意领地已定义（核心信息+情感氛围）
- [ ] 至少已记录3条评估标准
- [ ] 护栏已记录（什么是禁区）
- [ ] 策略已获决策者批准
- [ ] 品牌项目：品牌定位和平台已撰写

→ 下一步：使用 **04-generate** 探索概念、原型和AI辅助视觉。

## 设计决策日志（重要决策必填）

所有重要的策略决策必须记录。每条日志不仅记录选择了什么，还记录考虑过什么以及为什么被拒绝。

**何时记录：**
- 从多个可行选项中选定创意领地
- 选择评估标准（为什么是这3-5条而不是其他）
- 解决冲突（链接到冲突解决块）
- 任何明确考虑并拒绝了合理替代方案的决策

**日志格式：**

```
决策：[决定了什么]

选项：
A. [替代方案——简要描述]
B. [替代方案——简要描述]
C. [选定方案——简要描述]

选定：C

拒绝：
A → [为什么——与策略或用户需求挂钩的具体理由]
B → [为什么——与策略或用户需求挂钩的具体理由]

理由：
→ [为什么C最有利于项目目标——与设计简报和成功标准挂钩]
```

**规则：**
- "我就是知道"不是理由。每个决策必须追溯到一个策略原因。
- 日志条目是 07-learn 审计中的证据。重要决策缺少日志 = 审计中的 ⚠。
- 精简模式：至少记录一条主要的策略决策。标准/深度模式：记录所有重要决策。
