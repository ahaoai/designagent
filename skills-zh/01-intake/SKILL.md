# 01-Intake：项目导入

## 概述
每个设计项目都从导入开始——不是调研，不是画草图，而是理解。在去发现任何东西之前，你需要知道：我们在做什么，为谁而做，成功长什么样？此技能获取简报、约束和利益相关者信息。产出直接输入02-discover。

## 不可协商的规则
**硬性门槛：在进入任何设计阶段前完成导入。** 如果你说不清客户是谁、交付物是什么、截止日期是什么时候，你就还没有准备好设计。不做调研，不出方案，不画草图——只是理解。

## 何时使用
- 启动任何新设计项目
- 新请求到来，但范围、受众或交付物不清晰
- 在任何调研、画草图或概念工作之前

## 何时不使用
- "打开我的 Figma 文件" → 直接操作
- "LOGO 是什么颜色？" → 直接回答，这是查找
- "把这个图片改小" → 这是生产，不是设计
- 正在进行的已有项目的修改 → 从上次中断处继续

## 流程

### 1. 捕获简报
- 项目是什么？（类型、范围、类别）
- 交付物是什么？（列出具体产出）
- 时间线如何？（关键里程碑、硬截止日期）
- 预算或资源情况如何？（如已知）

### 2. 识别利益相关者
- 客户/决策者是谁？
- 最终用户是谁？
- 还需要谁来批准？
- 谁来实施/开发/生产？

### 3. 列出约束
- 品牌约束（现有指南、品牌资产）
- 平台/媒介约束（数字、印刷、物理）
- 技术/生产约束
- 法规或法律约束
- 文化或市场约束

### 4. 定义成功标准
- 什么结果最重要？（业务、用户、创意）
- 成功如何衡量？
- "完成"长什么样？
- 客户有哪些不可协商的要求？

### 5. 评估项目类型
- 选择DA工作流路径：
  - 品牌/识别 → 包含 brand-strategy
  - 室内/建筑 → 包含 interior-design 或 architecture-design
  - UI/UX/产品 → 包含 ui-ux-design
  - 工业/产品设计 → 包含 industrial-design
  - 通用设计 → 仅使用核心A层

## 理性化预防

| 借口 | 事实 |
|------|------|
| "我已经了解项目了，开始吧" | 写下来。导入把隐含假设变得明确。 |
| "成功标准可以后面再定义" | 现在就定义成功，否则你都不知道什么时候算完。 |
| "简报很简单，这太过了" | 简单的简报正是假设藏身的地方。暴露它们。 |

## 危险信号
- 你跳过了导入并开始画草图
- 你说不出决策者是谁
- 成功标准很模糊（"好看就行"）
- 时间线没有里程碑

## 验证
- [ ] 项目简报已用一段话记录
- [ ] 关键利益相关者已识别并注明角色
- [ ] 至少已记录3项约束
- [ ] 成功标准已定义（不只是"好看"）
- [ ] 已选择DA工作流路径
- [ ] 进行下一阶段前已确认
