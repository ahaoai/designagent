# 06-Deliver：提案、开发与交付

## 概述
交付不仅仅是发送文件。它是确保设计能够被正确构建、实施或生产。此技能涵盖最终资产生产、规格编写、交付文档和最终客户提案。好的交付让下一个人成功，而不需要你在房间里。

## 不可协商的规则
**硬性门槛：任何文件在离开你之前必须完成交付清单。** "他们知道该怎么做"不是交付。写下来。

## 何时使用
- 评审完成且设计冻结后
- 准备好产出最终文件、规格和交付物时

## 何时不使用
- 评审之前（设计必须先冻结——回到 05）
- 还在做创意决策时（回到 04 或 05）
- "先给他们快速导出"——没有规格的快速导出制造下游问题

## 流程

### 1. 定稿生产资产
- 以正确规格导出所有最终文件（格式、分辨率、色彩配置）
- 用清晰的命名规范组织文件
- 在适当时包含源文件
- 检查：字体、图片和资产是否都已打包？

### 2. 编写规格
- 技术规格：尺寸、文件格式、色彩值、字体
- 行为规格（数字类）：状态、交互、断点、动画
- 生产规格（印刷类）：出血、边距、材料、表面处理
- 实施说明：开发者或生产者需要知道的

### 3. 准备交付文档
- 资产清单：包含什么、在哪里
- 使用指南：如何应用、扩展或修改
- 响应式行为（数字类）：如何适配
- 已知限制或边缘情况

### 4. 构建最终提案或演示
如需向客户或利益相关者展示，调用 **design-narrative**：
- 从问题和策略开始
- 展示旅程（探索了什么、拒绝了什么、为什么）
- 将最终方案呈现为水到渠成
- 预判异议
- 以下一步结束，不以开放问题结束

### 5. 交付和归档
- 使用README或交付文档交付文件
- 归档项目材料（源文件、调研、简报、版本）
- 确认接收方已收到并理解

## 理性化预防

| 借口 | 事实 |
|------|------|
| "我先发文件，后面再解释" | 你不会的。写下交付文档。 |
| "开发者知道该怎么做" | 他们知道自己的工作。你的设计决策并非显而易见。 |
| "客户说他们信任我" | 信任不能替代规格。还是写下来。 |
| "发送前再调一下" | 完成好过完美。交付并进入验证。 |

## 危险信号
- 文件命名为"最终版_v3_最终版.ai"
- 没有书面规格存在
- 开发者或生产者未确认收到
- 你在做应该在评审阶段捕获的修改

## 验证
- [ ] 所有生产资产按正确规格导出
- [ ] 书面规格完成（技术、行为、生产）
- [ ] 交付文档包含资产清单和使用指南
- [ ] 最终提案或演示已准备（如需）
- [ ] 文件已交付并确认接收
- [ ] 项目材料已归档
